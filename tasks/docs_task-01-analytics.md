# Задача 1. Аналитика: Требования к MVP

## 1. Цель задачи
Выделение минимально жизнеспособного продукта (MVP) для приложения записи в гончарную мастерскую «Глина» на основе сырого и противоречивого письма заказчицы Марины. Цель состоит в формализации ролей, пользовательских историй (User Stories), определении жестких границ скоупа и описании ключевых сценариев использования (Use Cases).

## 2. Ссылка на бизнес-требования
* **Источник:** Письмо Марины (`brief-pottery.md`) + Уточнения по скоупу (инварианты интеграции бэкенда).
* **Бизнес-правила:**
  * **R-027:** Дефолтный горизонт расписания — строго 7 дней от текущей даты. Наличие Empty State.
  * **R-cap:** Вместимость групп: гончарный круг — строго до 6 человек, лепка руками — до 10 человек.
  * **R-rent:** Опция проката инструментов и фартука предоставляется без указания размеров.
  * **R-004:** Защита от двойных броней через атомарную проверку мест на бэкенде (возврат 409 Conflict).
  * **R-008:** Запрет записи на занятия со статусом `CANCELLED_BY_STUDIO` с отображением причины отмены.

## 3. Реализация и описание требований
Выделены две ключевые роли: **Клиент** (телефонный интерфейс для поиска слотов и записи) и **Владелец/Администратор** (панель управления расписанием). Сформулированы эпики:
* **Эпик A. Каталог расписания** (US-A1...US-A4): вывод слотов на 7 дней, визуальное отличие полных/отмененных слотов, обработка пустых состояний.
* **Эпик B. Карточка занятия** (US-B1...US-B3): детализация программы, стоимости, проката, причин форс-мажора.
* **Эпик C. Бронирование** (US-C1...US-C4): атомарная запись по имени и телефону, аренда инструментов.
* **Эпик D. Мои записи** (US-D1...US-D2): поиск по телефону, ограничение поздних отмен (менее чем за 2 часа).
* **Эпик E. Мастера** (US-E1): просмотр команды и их рейтингов.
* **Эпик F. Администрирование** (US-F1...US-F2): ручная правка слотов, перенос времени, отмена занятий.

## 4. Отправленные промпты
1. `Роль: Senior Full-Stack разработчик и системный аналитик. Проект — MVP клиентского приложения записи в гончарную мастерскую «Глина». Вот бриф заказчика (письмо + уточнения по скоупу): <текст brief-pottery.md>. Проанализируй бриф и определи границы MVP. Выдай результат чистым Markdown.`
2. `На основе брифа сгенерируй требования к MVP: роли, User Stories по эпикам (Каталог, Карточка, Бронирование) и ключевые сценарии использования (happy path + краевые: заполненный слот, отменённый слот, пустое расписание). Учитывай бизнес-правила: горизонт 7 дней (R-027), вместимость 6/10, прокат без размеров, атомарная проверка мест с 409 (R-004), запрет записи на CANCELLED_BY_STUDIO (R-008). Добавь критерии приёмки.`
3. `Из письма и уточнений выпиши таблицу бизнес-правил с ID (R-027, R-004, R-008 и т.д.) — по одному правилу на строку, кратко и однозначно.`
4. `Добавь в требования сценарии, прямо упомянутые в письме: отмена брони клиентом с освобождением места (но с ограничением поздних отмен), витрина мастеров с рейтингами и экран владельца, где видно всё расписание и можно «поправить руками».`

# Задача 2. Проектирование: Архитектура и Схема данных

## 1. Цель задачи
Разработка архитектурного плана приложения и проектирование канонической схемы данных (контракта REST API), обеспечивающей выполнение бизнес-правил (включая атомарность транзакций бронирования и бесконфликтность данных).

## 2. Ссылка на бизнес-требования
* **R-004 (Атомарность):** Перенос всей логики валидации мест на бэкенд, выполняющий роль black-box источника истины.
* **R-015 (Каноническая схема данных):** Бэкенд должен отдавать все поля контракта (тарифы проката, описания программ, адрес студии, доступный прокатный фонд).
* **Связанные правила:** R-027 (фильтрация дат на сервере), R-008 (сохранение истории отмененных слотов).

## 3. Реализация архитектурных решений
Выбрана клиент-серверная архитектура с полным разделением ответственности:
* **Frontend:** 7 статических экранов (`index.html`, `details.html`, `success.html`, `masters.html`, `bookings.html`, `login.html`, `admin.html`), взаимодействующих с API через асинхронный `fetch()`. Статические ресурсы обслуживаются независимым сервером.
* **Backend REST API:** Изолированное приложение на Python/Flask. Хранилище данных — in-memory словари (`SLOTS`, `BOOKINGS`, `PROGRAMS`, `MASTERS`). 
* **Обеспечение транзакционности:** Критическая секция метода `create_booking` и `cancel_booking` защищена глобальным блокировщиком `threading.Lock()`. Это полностью исключает race condition (состояние гонки) при конкурентных запросах на последнее свободное место.
* **Схема данных:** Спроектированы сущности `Program`, `Master`, `Slot`, `Booking` и `User/Token` с явным указанием типов данных и примеров ответов.

# Задача 2. Проектирование: архитектура и схема данных

> Архитектурный план и контракт API, полученные с ИИ.
> Требования — `docs/01-requirements.md`, реализация — `docs/03-features.md`.

## 1. Архитектура

Клиент-серверное приложение из двух независимых частей:

```text
┌──────────────────────────┐        HTTP / JSON        ┌──────────────────────────┐
│  Frontend (browser)      │  ───────────────────────▶ │  Backend REST API        │
│  HTML / CSS / JS,        │  ◀─────────────────────── │  Python / Flask          │
│  без фреймворков         │        сырой JSON         │  in-memory «БД»          │
│  7 экранов + app.js      │                           │  словари SLOTS/BOOKINGS  │
│                          │   Authorization: Bearer   │  атомарное бронирование  │
└──────────────────────────┘                           └──────────────────────────┘
```

* **Backend** — изолированный REST API (`backend/api.py`). Хранилище имитируется
  in-memory словарями. Отдаёт **сырой JSON**. Отвечает за фильтрацию расписания,
  валидацию, **транзакционное** бронирование, аутентификацию и админ-операции.
* **Frontend** — 7 статических экранов + общий `app.js`, ходит в API через `fetch`.
* Связь закрыта **контрактом API**. Бэкенд — black-box источник истины; гарантия
  «0 двойных броней» на его стороне (R-004).

### Экраны фронтенда
`index.html` (расписание) · `details.html` (карточка) · `success.html`
(подтверждение) · `masters.html` (мастера) · `bookings.html` (мои записи) ·
`login.html` (вход) · `admin.html` (панель администратора).

### Почему так

* Разделение фронта и API — требование задания, упрощает ручную проверку.
* In-memory хранилище достаточно для учебного MVP (легаси нет, R-015).
* Атомарность — один `threading.Lock` вокруг критической секции; минимально
  достаточно для защиты от гонки при бронировании и отмене.
* Аутентификация — простейшая токен-схема in-memory (учебная), роль проверяется
  на админ-роутах.

## 2. Модель данных (in-memory)

### Program (программа занятия)

| Поле         | Тип | Пример / примечание                       |
| ------------ | --- | ----------------------------------------- |
| id           | str | `"wheel"` / `"handbuilding"`              |
| code         | str | `WHEEL` / `HANDBUILDING`                  |
| title        | str | «Гончарный круг для новичков»             |
| description  | str | текст программы                           |
| max_capacity | int | **6** (круг) / **10** (лепка)             |
| price        | int | цена, ₽                                   |

### Master (мастер)

| Поле          | Тип   | Примечание       |
| ------------- | ----- | ---------------- |
| id            | str   | `"m1"`           |
| name          | str   | «Анна»           |
| rating        | float | 4.9              |
| ratings_count | int   | 132              |
| specialty     | str   | «Гончарный круг» |
| bio           | str   | биография        |

### Slot (слот расписания) — каноническая сущность

| Поле             | Тип       | Примечание                                   |
| ---------------- | --------- | -------------------------------------------- |
| id               | str       | `"s1"`                                       |
| program_id       | str       | → Program                                    |
| master_id        | str       | → Master                                     |
| start_time       | str (ISO) | `"2026-07-05T11:00"`                         |
| duration_min     | int       | 120–150 (2–2.5 ч)                            |
| capacity         | int       | вместимость слота (≤ max_capacity программы) |
| booked_count     | int       | сколько уже занято                           |
| status           | str       | `SCHEDULED` / `CANCELLED_BY_STUDIO`          |
| cancel_reason    | str/null  | причина отмены (R-008)                       |
| rental_available | bool      | доступен ли прокат                           |
| rental_price     | int       | цена проката (без размеров)                  |
| rental_stock     | int       | свободный прокатный фонд (R-015)             |

Вычисляемые поля в ответе API: `remaining_places = capacity − booked_count`,
`is_bookable = status == SCHEDULED AND remaining_places > 0`.

### Booking (бронь)

| Поле           | Тип       | Примечание                          |
| -------------- | --------- | ----------------------------------- |
| id             | str       | генерируется                        |
| slot_id        | str       | → Slot                              |
| customer_name  | str       | обязательно                         |
| customer_phone | str       | обязательно                         |
| rental         | bool      | взял ли прокат                      |
| status         | str       | `CONFIRMED` / `CANCELLED_BY_CLIENT` |
| created_at     | str (ISO) | момент брони                        |

### User / Token (аутентификация, учебная схема)

| Поле        | Тип | Примечание                         |
| ----------- | --- | ---------------------------------- |
| login       | str | `"admin"` / `"client"`             |
| password    | str | in-memory, открытым текстом (демо) |
| role        | str | `admin` / `client`                 |
| name, phone | str | профиль; телефон — у клиента       |

Токены — словарь `token → login`, передаются в `Authorization: Bearer <token>`.

## 3. Контракт API

Базовый префикс: `/api`. Все ответы — JSON. CORS открыт (`*`); разрешённые
заголовки — `Content-Type, Authorization` (последний обязателен для админ-роутов).

### Каталог и карточка

| Метод / путь          | Назначение                                                                               | Коды      |
| --------------------- | ---------------------------------------------------------------------------------------- | --------- |
| `GET /api/slots`      | каталог; фильтр `date_from` / `date_to`, дефолт **+7 дней** (R-027); прошедшие исключены | 200       |
| `GET /api/slots/{id}` | карточка занятия                                                                         | 200 / 404 |
| `GET /api/masters`    | команда с рейтингами (сорт. по рейтингу)                                                 | 200       |
| `GET /api/health`     | живость (`{status, slots}`)                                                              | 200       |

**Пример `GET /api/slots` → 200 OK:**
```json
{
  "count": 1,
  "slots": [
    {
      "id": "s1", "start_time": "2026-07-05T11:00", "duration_min": 150,
      "status": "SCHEDULED", "cancel_reason": null,
      "capacity": 6, "booked_count": 2, "remaining_places": 4,
      "is_bookable": true,
      "program": { "code": "WHEEL", "title": "Гончарный круг для новичков",
                   "description": "…", "max_capacity": 6, "price": 2500 },
      "master": { "name": "Анна", "rating": 4.9, "ratings_count": 132,
                  "specialty": "Гончарный круг" },
      "rental_available": true, "rental_price": 400, "rental_stock": 8,
      "studio_address": "г. Москва, ул. Гончарная, д. 12, …"
    }
  ]
}
```

### Бронирование

| Метод / путь                     | Назначение                                | Коды                      |
| -------------------------------- | ----------------------------------------- | ------------------------- |
| `POST /api/bookings`             | создать бронь                             | 201 / 400 / 404 / **409** |
| `GET /api/bookings?phone=…`      | мои записи по телефону (норм. до 10 цифр) | 200 / 400                 |
| `GET /api/bookings/{id}`         | одна бронь                                | 200 / 404                 |
| `POST /api/bookings/{id}/cancel` | отмена клиентом (правило 2 ч)             | 200 / **409** |

`409` при бронировании: `no_places` (нет мест, R-004) **или** `slot_cancelled`
(слот отменён, R-008). `409` при отмене: `late_cancellation` (< 2 ч),
`already_cancelled`, `slot_cancelled`.

### Аутентификация

| Метод / путь           | Назначение                            | Коды      |
| ---------------------- | ------------------------------------- | --------- |
| `POST /api/auth/login` | `{login, password}` → `{token, user}` | 200 / 401 |
| `GET /api/auth/me`     | профиль по токену                     | 200 / 401 |

### Администрирование (роль `admin`, иначе 403)

| Метод / путь                         | Назначение                                                           | Коды                        |
| ------------------------------------ | -------------------------------------------------------------------- | --------------------------- |
| `GET /api/admin/slots`               | всё расписание (вкл. прошедшие/отменённые) + состав броней           | 200 / 403                   |
| `POST /api/admin/slots/{id}/cancel`  | отменить занятие `{reason}` (R-008)                                  | 200 / 403 / 404             |
| `POST /api/admin/slots/{id}/restore` | вернуть занятие в расписание                                         | 200 / 403 / 404             |
| `PATCH /api/admin/slots/{id}`        | ручная правка: `capacity`, `master_id`, `start_time`, `duration_min` | 200 / 400 / 403 / 404 / 409 |

Валидация `PATCH`: `capacity` — не ниже уже записанных и не выше максимума
программы (иначе 409); `master_id` — должен существовать (иначе 400); `start_time`
— корректный ISO (иначе 400); `duration_min` — 30–300 мин (иначе 400).

## 4. Транзакционность бронирования и отмены (R-004)

Критическая секция «проверить → изменить» выполняется под глобальным
`threading.Lock`, чтобы два параллельных запроса не переполнили слот:

```python
with _booking_lock:
    slot = SLOTS[slot_id]
    if slot.status == CANCELLED_BY_STUDIO:   # R-008
        return 409
    if remaining_places <= 0:                # R-004
        return 409
    slot.booked_count += 1
    BOOKINGS[id] = {...}
```

Отмена брони использует тот же замок: освобождение места (`booked_count -= 1`)
атомарно, с проверкой правила отсечки за 2 часа.

## 5. Маршрут пользователя (frontend)

```text
index.html ──выбор слота──▶ details.html ──POST /bookings 201──▶ success.html
   (GET /slots)               (GET /slots/{id})                    │
        │                                                          ▼
        └──────────────── bookings.html (GET /bookings?phone=…) ◀──┘
 login.html ──роль admin──▶ admin.html (GET/POST/PATCH /admin/slots)
```

## Приложение. Использованные промпты

**Промпт 1 (архитектура):**
> На основе требований (`01-requirements.md`) предложи архитектуру MVP: независимый
> REST API на Python/Flask с in-memory хранилищем и фронтенд на чистом HTML/CSS/JS.
> Нарисуй схему связей ASCII, объясни ключевые решения. Бэкенд — black-box источник
> истины, гарантия отсутствия двойных броней — атомарная проверка на сервере
> (R-004). Формат — чистый Markdown.

**Промпт 2 (схема данных):**
> Спроектируй модель данных для in-memory хранилища: Program, Master, Slot,
> Booking, User/Token. Для Slot учти статус CANCELLED_BY_STUDIO и причину отмены,
> вместимость (круг 6 / лепка 10), прокат без размеров и прокатный фонд. Выдай
> таблицы полей с типами и примерами.

**Промпт 3 (контракт API):**
> Опиши контракт REST API: каталог/карточка, бронирование, мои записи и отмена,
> аутентификация, админ-роуты (просмотр всех слотов, отмена/восстановление, PATCH
> с правкой capacity/master_id/start_time/duration_min). Для каждого эндпоинта —
> параметры, примеры JSON и полный список кодов. Явно опиши 409 при отсутствии мест
> (R-004) и при отменённом слоте (R-008).

**Промпт 4 (транзакционность):**
> Покажи псевдокод атомарной секции под threading.Lock: сначала проверка статуса
> слота (R-008), затем число мест (R-004), потом инкремент booked_count и запись
> брони. Отмену брони реализуй под тем же замком.

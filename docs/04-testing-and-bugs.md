## 2. Реестр багов (#1–#10)

### Баг #1 — переполнение слота: бронь заполненного занятия проходит (логический)

* **Симптом:** бронь слота с `remaining_places = 0` возвращала `201` и уводила
  `booked_count` за `capacity` (двойная бронь).
* **Требование:** R-004 — атомарная проверка, при отсутствии мест → 409.
* **Причина:** проверка использовала `remaining < 0` вместо `remaining <= 0`.
* **Фикс:** условие изменено на `<= 0` (`backend/api.py`, `create_booking`).
* **Тест-кейсы:** TC-10, TC-12.

### Баг #2 — запись на отменённое мастерской занятие (логический)

* **Симптом:** бронь слота со статусом `CANCELLED_BY_STUDIO` возвращала `201`.
* **Требование:** R-008 — запись на отменённый слот запрещена.
* **Причина:** в `create_booking` не было проверки статуса перед созданием брони.
* **Фикс:** добавлена проверка `status == CANCELLED_BY_STUDIO` → `409 slot_cancelled`.
* **Тест-кейс:** TC-11.

**Единый диф по бэкенду (Баги #1 и #2):**
```diff
         slot = SLOTS.get(slot_id)
         if slot is None:
             return jsonify({"error": "slot not found"}), 404

+        # R-008: запрет записи на слот, отменённый мастерской.
+        if slot["status"] == STATUS_CANCELLED_BY_STUDIO:
+            return jsonify({
+                "error": "slot_cancelled",
+                "message": "Занятие отменено мастерской. Запись недоступна.",
+                "cancel_reason": slot["cancel_reason"],
+            }), 409
+
         remaining = slot["capacity"] - slot["booked_count"]
-        if remaining < 0:
+        # R-004: нет свободных мест — отклоняем бронь.
+        if remaining <= 0:
             return jsonify({
                 "error": "no_places", "message": "Мест не осталось",
-                "remaining_places": max(remaining, 0),
+                "remaining_places": 0,
             }), 409
```

### Баг #3 — тупиковая кнопка «Записаться» на заполненном слоте (UI)

* **Симптом:** полностью заполненный слот (`remaining = 0`, не отменён) показывал
  активную кнопку «Записаться», ведущую в карточку, где запись недоступна.
* **Требование:** US-A4 (визуально отличать заполненные занятия).
* **Причина:** `renderSlotCard` учитывал только флаг `cancelled`, но не `full`.
* **Фикс:** для заполненных (но не отменённых) слотов — неактивная кнопка «Мест
  нет» (`frontend/app.js`).
* **Тест-кейс:** TC-28.

### Баг #4 — дубль даты в «Моих записях» (UI)

* **Симптом:** «понедельник, 6 июля, 6 июля в 19:00» — дата дважды.
* **Причина:** конкатенация `dayLabel()` и `formatDateTime()` — обе содержат дату.
* **Фикс:** в карточке брони — `dayLabel()` + только время (`formatTime()`).
* **Тест-кейс:** TC-29.

### Баг #5 — фильтр «14 дней» не показывал новых занятий (данные)

* **Симптом:** переключение на 14 дней визуально ничего не добавляло.
* **Причина:** логика корректна, но в демо-данных за горизонтом 7 дней был всего
  один слот — разница незаметна.
* **Фикс:** расписание дополнено слотами на дальние дни (позже — переход на
  ежедневную генерацию, см. `03-features.md`).
* **Тест-кейс:** TC-05.

### Баг #6 — дубль даты в карточке занятия и на экране успеха (UI)

* **Симптом:** «пятница, 10 июля · 10 июля в 18:00» — тот же класс ошибки, что #4,
  но в двух других местах.
* **Причина:** конкатенация двух функций, каждая из которых содержит дату.
* **Фикс:** введена единая `formatWhen()` («Сегодня, 4 июля в 18:00»); дублирующая
  `formatDateTime()` удалена.
* **Тест-кейс:** TC-29.

### Баг #7 — капитализация месяцев через CSS (UI/типографика)

* **Симптом:** «Пятница, 10 Июля» — в русском названия месяцев со строчной.
* **Причина:** `text-transform: capitalize` капитализирует **каждое** слово.
* **Фикс:** `text-transform` убран; первая буква дня недели капитализируется в
  `dayLabel()` на стороне JS, месяц остаётся со строчной.
* **Тест-кейс:** TC-30.

### Баг #8 — сводка админки наследовала десктопные стили hero (UI/вёрстка)

* **Симптом:** на ≥ 900px сводка «Слотов / Записей / Отменено» разъезжалась в
  вертикальную карточку.
* **Причина:** блок переиспользовал класс `hero__stats`, у которого в десктопном
  брейкпоинте заданы стили колонки-карточки внутри hero.
* **Фикс:** для сводки выделен собственный класс `.stats-row` (горизонтальный ряд).
* **Тест-кейс:** TC-31.

### Баг #9 — панель администратора не грузила расписание (CORS, блокирующий)

* **Симптом:** «Не удалось загрузить расписание. Проверьте сервер», хотя сервер
  работал и `/api/admin/slots` напрямую отвечал `200`; публичное `/api/slots`
  грузилось нормально.
* **Причина:** заголовок `Access-Control-Allow-Headers` разрешал только
  `Content-Type`. Запросы админки шлют `Authorization: Bearer <token>`, поэтому
  браузер блокировал их на этапе CORS-preflight (`OPTIONS`) — до отправки; `fetch`
  падал с сетевой ошибкой. Публичный `/slots` авторизации не требует — работал.
* **Фикс** (`backend/api.py`, `add_cors_headers`):
  ```diff
  -    response.headers["Access-Control-Allow-Methods"] = "GET, POST, OPTIONS"
  -    response.headers["Access-Control-Allow-Headers"] = "Content-Type"
  +    response.headers["Access-Control-Allow-Methods"] = "GET, POST, PATCH, OPTIONS"
  +    response.headers["Access-Control-Allow-Headers"] = "Content-Type, Authorization"
  ```
* **Тест-кейсы:** TC-21, TC-32.

### Баг #10 — бесполезная кнопка-переключатель формы правки (UI/CSS)

* **Симптом:** кнопка «Перенести · сменить мастера» «ничего не делала» — форма
  правки была видна всегда, нажатие визуально ни на что не влияло.
* **Причина:** кнопка переключала HTML-атрибут `hidden`, но правило CSS
  `.admin-row__edit { display: flex }` перебивает браузерное `[hidden] { display:
  none }` (у `display` выше приоритет), поэтому форма оставалась видимой.
* **Фикс:** переключатель убран, форма правки показывается постоянно под каждой
  строкой (с заголовком «Правка вручную:»); удалена ненужная кнопка «Отмена».
* **Тест-кейс:** TC-33.

**Ключевые коммиты багфиксов:**
* `fix: запрет брони заполненных и отменённых слотов (R-004, R-008) + UI`

## Приложение. Использованные промпты

**Промпт 1 (генерация тест-кейсов):**
> На основе контракта API сгенерируй тест-кейсы для всех эндпоинтов (каталог,
> карточка, бронирование, мои записи/отмена, аутентификация, админ), включая
> краевые: заполненный слот, отменённый слот, пустое расписание, поздняя отмена.
> Добавь UI-кейсы. Оформи таблицами: сценарий → ожидание → результат.

**Промпт 2 (поиск логических багов):**
> Прогони TC-10 и TC-11 через Flask test client. Почему бронь заполненного и
> отменённого слота возвращает 201 вместо 409? Найди причину в create_booking и
> исправь под R-004 и R-008. Дай единый диф.

**Промпт 3 (UI-баги):**
> Заполненный слот показывает активную кнопку «Записаться» в тупик; в «Моих
> записях», карточке и на экране успеха дата выводится дважды; месяцы пишутся с
> заглавной; сводка админки разъезжается на десктопе. Для каждого бага — симптом,
> причина, фикс.

**Промпт 4 (блокирующий баг CORS и чистка):**
> Панель администратора выдаёт «Не удалось загрузить расписание», хотя API отвечает
> 200 напрямую. Найди причину (подсказка: preflight и Authorization) и исправь.
> Затем убери бесполезную кнопку, которая ничего не делает, и объедини все баги в
> один документ.

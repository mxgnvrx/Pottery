# Спецификация тестового задания: MVP приложения «Гончарная мастерская»

## 1. Требования к заданию (Основные задачи)
Цель задания — продемонстрировать навыки работы с ИИ в качестве напарника (AI-driven development) для полного цикла создания продукта: от анализа требований до готового кода и тестирования.

В рамках выбранного брифа («Гончарная мастерская») необходимо выполнить 4 этапа:
1. **Аналитика:** Сгенерировать с ИИ требования к MVP (User Stories + основные сценарии использования).
2. **Проектирование:** Получить от ИИ архитектурный план и схему данных (контракты API).
3. **Разработка:** Реализовать минимум 3 ключевые фичи (Каталог, Карточка занятия, Бронирование).
4. **Тестирование и Багфикс:** Сгенерировать тест-кейсы, найти и исправить от 1 до 3 логических или UI-багов.

## 2. Требования к выполнению (Процесс)
Процесс выполнения строго регламентирован для прозрачности взаимодействия разработчика и ИИ:
* **Документирование каждого шага:** Для каждой задачи и каждого найденного бага должен быть заведен отдельный `.md` документ.
* **Структура отчетов:** В каждом `.md` документе необходимо:
  * Кратко описать цель задачи или симптом бага.
  * Указать ссылку на бизнес-требование из исходного брифа (если применимо).
  * Описать реализацию или доработку.
  * Обязательно **прикрепить все отправленные промпты**, которые привели к результату.
* **Ручной контроль:** Итоговый код и решения ИИ должны быть проверены вручную.
* **Версионирование:** Каждая решенная задача должна сопровождаться отдельным Git-коммитом.

## 3. Требования к бизнес-логике (Специфика брифа)
Реализация должна строго соответствовать вводным данным заказчика:
* **Каталог:** Дефолтный горизонт расписания — строго 7 дней (R-027). Наличие Empty State при пустом расписании.
* **Вместимость:** Занятия на гончарном круге для новичков вмещают строго до 6 человек. Обычная лепка — до 10 человек.
* **Прокат:** Опция проката инструментов и фартука предоставляется без указания размеров (в отличие от других брифов).
* **Бронирование:** Гарантия защиты от «двойных броней» (R-004) — атомарная проверка мест на бэкенде.
* **Отмены:** Запрет записи на занятия, отмененные по форс-мажору мастерской (статус `CANCELLED_BY_STUDIO`).

## 4. Требуемые результаты (Артефакты)
На выходе должен получиться готовый репозиторий со следующей структурой:
1. **Документация (папка `docs/`):** 4 файла `.md` с отчетами о генерации требований, архитектуры, фич и багфиксов (с прикрепленными промптами).
2. **Бэкенд (папка `backend/`):** Изолированный REST API на Python/Flask, реализующий логику валидации, фильтрации расписания и транзакционного бронирования слотов. Возвращает сырой JSON.
3. **Фронтенд (папка `frontend/`):** Веб-приложение (HTML/CSS/JS), работающее с API. Включает 3 экрана:
   * `index.html` — Расписание занятий.
   * `details.html` — Детали слота и форма записи.
   * `success.html` — Экран успешного бронирования.
4. **История Git:** Минимум 4 осмысленных коммита, отражающих поэтапное выполнение задания.

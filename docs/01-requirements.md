# Задача 1. Аналитика: требования к MVP

> Генерация требований к MVP с ИИ (User Stories + основные сценарии).
> Источник — `docs/brief-pottery.md` (письмо заказчицы Марины + уточнения по
> скоупу). Архитектура — `docs/02-architecture.md`, реализация —
> `docs/03-features.md`.

## 1. Цель

Из сырого, местами противоречивого письма заказчицы выделить **минимальный
жизнеспособный продукт (MVP)** приложения записи в гончарную мастерскую «Глина»:
сформулировать User Stories и ключевые сценарии использования.

## 2. Роли

* **Клиент** — основная роль поставки (R-028): пользуется с телефона, ищет
  свободные слоты и записывается.
* **Владелец / администратор** — по отдельному пожеланию из письма Марины («экран,
  где я вижу всё расписание и могу что-то поправить руками, если что-то пошло не
  так»). Добавлен как второй интерфейс поверх базового MVP.
* *(Мастер — существует в инфраструктуре, вне скоупа демо.)*

## 3. Границы MVP (что делаем / что нет)

| В скоупе                                   | Вне скоупа (отложено / чужая зона)       |
| ------------------------------------------ | ---------------------------------------- |
| Просмотр расписания (дефолт 7 дней, R-027) | Онлайн-оплата (пока наличные/перевод)    |
| Карточка занятия с деталями и прокатом     | Выставление оценки мастеру после занятия |
| Бронирование места + прокат                | Push-уведомления (упомянуты, не в MVP)   |
| Обработка отказа «нет мест» (409, R-004)   | Программа лояльности                     |
| Запрет записи на отменённые слоты (R-008)  | Полноценный интерфейс мастера            |
| «Мои записи» + отмена брони клиентом       |                                          |
| Витрина мастеров с рейтингами              |                                          |
| Панель администратора (управление слотами) |                                          |

## 4. User Stories

### Эпик A. Каталог расписания

* **US-A1.** Как клиент, я хочу видеть список ближайших занятий на **7 дней вперёд**
  (R-027), чтобы быстро выбрать подходящее время.
* **US-A2.** Как клиент, я хочу видеть по каждому слоту программу, дату/время,
  мастера и число свободных мест, чтобы принять решение без лишних кликов.
* **US-A3.** Как клиент, если занятий нет, я хочу видеть понятное сообщение
  **empty state** («Пока нет доступных занятий»), а не пустой экран.
* **US-A4.** Как клиент, я хочу отличать заполненные и отменённые занятия от
  свободных по визуальному признаку (бейдж/цвет).

### Эпик B. Карточка занятия

* **US-B1.** Как клиент, я хочу открыть карточку слота и увидеть описание
  программы, длительность, вместимость, рейтинг мастера и адрес мастерской.
* **US-B2.** Как клиент, я хочу видеть, доступен ли **прокат инструментов и фартука**
  и сколько он стоит (без выбора размера — специфика брифа).
* **US-B3.** Как клиент, если занятие отменено мастерской, я хочу видеть причину
  отмены и понимать, что запись недоступна (R-008).

### Эпик C. Бронирование

* **US-C1.** Как клиент, я хочу записаться на свободный слот, указав имя и телефон.
* **US-C2.** Как клиент, я хочу опционально отметить, что беру инструменты и фартук
  напрокат.
* **US-C3.** Как клиент, я хочу получить экран подтверждения записи с деталями.
* **US-C4.** Как клиент, если пока я оформлял бронь места закончились, я хочу
  увидеть понятную ошибку, а не «молчаливый» провал (R-004, ответ 409).

### Эпик D. Мои записи

* **US-D1.** Как клиент, я хочу найти свои записи по номеру телефона.
* **US-D2.** Как клиент, я хочу отменить запись заранее и освободить место; поздняя
  отмена (менее чем за 2 часа) должна блокироваться вежливым сообщением.

### Эпик E. Мастера

* **US-E1.** Как клиент, я хочу видеть команду мастеров с рейтингами и
  специализацией, чтобы выбрать занятие «к звезде».

### Эпик F. Администрирование (владелец)

* **US-F1.** Как владелец, я хочу видеть **всё** расписание, включая прошедшие и
  отменённые занятия, с составом записавшихся.
* **US-F2.** Как владелец, я хочу **поправить занятие руками**, если что-то пошло
  не так: перенести время, заменить мастера, изменить число мест, отменить или
  вернуть занятие в расписание.

## 5. Ключевые сценарии использования (Use Cases)

### UC-1. Просмотр расписания и запись (happy path)
1. Клиент открывает приложение → видит занятия, сгруппированные по дням.
2. Выбирает «Гончарный круг, свободно 4 из 6».
3. Открывает карточку, читает детали, отмечает «прокат фартука».
4. Вводит имя и телефон, нажимает «Подтвердить запись».
5. Система атомарно резервирует место → экран успеха с деталями.

### UC-2. Запись на заполненный слот (защита от двойных броней)
1. Клиент открывает карточку слота, где остаётся 1 место.
2. Пока он заполняет форму, это место занимает другой клиент.
3. Клиент нажимает «Подтвердить» → бэкенд возвращает **409 Conflict**.
4. Приложение показывает «Мест не осталось», не создавая ложную бронь (R-004).

### UC-3. Отменённое мастерской занятие
1. У мастерской форс-мажор (сломалась печь) → слот получает статус
   `CANCELLED_BY_STUDIO`.
2. Клиент видит слот в списке с бейджем «Отменено мастерской».
3. В карточке отображается причина; форма записи скрыта (R-008).

### UC-4. Пустое расписание
1. На выбранный период занятий нет.
2. Клиент видит empty state вместо пустого экрана (US-A3).

### UC-5. Отмена брони клиентом
1. Клиент открывает «Мои записи» по телефону.
2. Отменяет будущую бронь (до начала > 2 часов) → место освобождается.
3. При попытке отменить менее чем за 2 часа → вежливый отказ «позвоните нам».

### UC-6. Правка расписания владельцем
1. Владелец входит в панель администратора.
2. Заболел мастер → владелец **заменяет мастера** в занятии.
3. Нужно перенести занятие → правит **дату/время**; при форс-мажоре — **отменяет**
   занятие с причиной (клиенты видят статус и причину).

## 6. Бизнес-правила (сведены из брифа)

| ID       | Правило                                                                                                             |
| -------- | ------------------------------------------------------------------------------------------------------------------- |
| R-027    | Дефолтный горизонт расписания — **7 дней** от текущей даты; длиннее — через фильтр; empty state при пустом периоде. |
| R-cap    | Круг для новичков — до **6** человек; лепка — до **10** человек.                                                    |
| R-rent   | Прокат инструментов и фартука — **без размеров**.                                                                   |
| R-004    | **Атомарная** проверка мест; нельзя забронировать при `remaining <= 0` → **409**.                                   |
| R-008    | Запрет записи на слоты со статусом `CANCELLED_BY_STUDIO`; при отмене видна причина.                                 |
| R-015    | Каноническая схема данных = контракт API; все поля отдаёт бэкенд.                                                   |
| R-028    | Базовая поставка — роль Клиент; роль владельца добавлена отдельным интерфейсом.                                     |
| R-cancel | Отмена брони клиентом — не позднее чем за **2 часа** до начала.                                                     |
| R-daily  | Студия работает **ежедневно**, по 3 занятия в день (уточнение заказчика).                                           |

## 7. Критерии приёмки (Definition of Done)

* Расписание по умолчанию показывает слоты за 7 дней; студия работает ежедневно;
  пустой период → empty state.
* Карточка отражает все поля контракта, включая прокат и адрес.
* Бронь свободного слота → 201 и экран успеха; заполненного → 409 без изменений;
  отменённого → отклонена, форма недоступна.
* Клиент может найти и отменить свои записи с учётом правила 2 часов.
* Владелец видит всё расписание и может править занятия вручную.
